Agent Observation Hub

這是 Pi coding agent 的觀測平台。每次你跟 Pi 對話，整個內部組裝過程（讀取 skills、組 system prompt、呼叫 LLM、執行 tool、拿結果再呼叫一次 LLM）都會被記錄在這裡，讓你可以「看到模型實際看到什麼」，而不是只看到回應。

第一次來？看這裡

名詞速查：

  • trace：一次 LLM 呼叫的完整紀錄（含 request / response / 延遲 / 用了多少 token）
  • session：一個 Pi agent 啟動後的整段互動（一個 session 通常包含很多 trace）
  • stage：agent 處理一個 user 訊息時經歷的步驟（例如載入 resources、組 context、呼叫模型、執行 tool）
  • provider adapter：pi-ai 套件裡負責把 agent 的內部訊息格式轉成各家 LLM API 認得格式的元件
  • pinned constraint：你 pin 起來的規則，會自動加到每一次 user prompt 的最前面

建議入門路徑：

  • 1️⃣ 看下方表格，找一個有 🔧 標記的 trace（代表那次有用 tool）
  • 2️⃣ 點進去，從上到下看 Agent Events 區塊就是完整的推理流程
  • 3️⃣ 找一個 Provider Adapter Diff 區塊，看看模型實際收到的內容跟 agent 內部存的差多少

最近的 LLM 呼叫紀錄 (Recent Trace Runs) ⓘ

每一個 row 代表 Pi 對 LLM 發起的一次呼叫。Stages 欄的圖示是這次呼叫所屬 session 的整體事件統計，幫你一眼看出哪些 session 有用 tool、改動多。

Stages 圖示說明
  • 👤 N 這個 session 收到過 N 次使用者輸入（before_agent_start 事件數）
  • 📚 這個 session 在啟動時載入過資源（skills / prompt templates / AGENTS.md）
  • 🤖 N 這個 session 對 LLM 發起過 N 次呼叫（一次使用者輸入可能觸發多次 LLM 呼叫，例如使用者問 → model 想用 tool → tool 跑完 → model 再看結果回答，這樣就是 2 次）
  • 🔧 N 這個 session 觸發過 N 次 tool 呼叫（read / bash / edit / write 等）
  • Δ N 這次 LLM 呼叫的 provider adapter 改了 N 個地方（context 跟 payload 不一致的程度）
