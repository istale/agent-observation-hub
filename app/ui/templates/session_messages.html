怎麼用這頁（第一次來必看）

把它想成「桌面整理」。所有標記都是視覺整理——原 JSONL 不會被改、訊息永遠還在那裡。差別只在「你眼睛和心思要不要分心去看」。

四個標記的詳細用法（什麼時候該用哪個？）

🟢 active（預設）

意思：「這條訊息現在還是主軸的一部分」

何時用：

  • 預設狀態，不用特別動
  • 之前被你標 background/stale 但又重新變成焦點 → 改回 active

例子：使用者問「幫我看 auth.py 的 bug」→ 整段討論都是 active

🟡 background（最常用）

意思：「這個對話分支現在不是重點，但還沒結束、之後可能還會回頭看」

何時用：

  • 多話題對話：你早上問了 A，下午轉去問 B，A 那段標 background
  • 不是錯的、不是過時，只是現在不關心
  • 像桌上的文件你還沒做完，但暫時推到旁邊

例子：早上 t1~t4 在討論 database schema → 下午轉成寫 API → 把 t1~t4 標 background。t1~t4 還在抽屜，你需要時打開來看；UI 變淡讓你眼睛專注在當下的 API 任務。

跟 stale 的差別：background = 「還活著、只是暫時不關心」；stale = 「死了/過時」

🔴 stale（明確標示過時/錯誤）

意思：「這條訊息已經不適用了，但我留著當紀錄 / 不想真的刪掉」

何時用：

  • 早期的決定被後面推翻了（model 做了錯的選擇、後面修正）
  • 你發現某個 tool 結果是錯的，後面才用對的方式
  • 某段討論基於錯誤前提，但你想保留證據

例子：

  • t2: assistant 說「用 vector DB」 → t5: 你說「不要 vector，用 grep」 → 把 t2 標 stale
  • t3: bash 跑錯路徑、結果不對 → 改為 stale 提醒自己這個 result 不可信

視覺上：紅底 + 刪除線 → 一眼看到「這條被否決過」

跟 background 的差別：background 是「暫時不看」；stale 是「永遠不適用了，但留著」

跟刪除的差別：你沒真的刪——之後想看 stale 訊息還是可以滑回去看，只是視覺上被否決掉

⚪ hidden（清掉視覺干擾）

意思：「這條訊息我根本不想看到，預設折疊起來」

何時用：

  • 又長又無聊（例如某個 tool 吐了 500 行 stdout、你都看過了）
  • 你已經消化過、不需要再看，但要保留證據
  • 大量重複資訊（例如連續 10 個 grep 結果）

例子：

  • t6 是 bash 吐了 300 行 file listing → hidden，眼睛不用受罪
  • t4 是一段 model 的長 thinking，已經看過 → hidden 折疊

視覺上：card 預設折疊，點開才看內容

跟 stale 的差別：stale 是「內容過時了」；hidden 是「內容沒過時但我看夠了」

跟 background 的差別：background 是「半透明還看得到」；hidden 是「徹底折疊」

快速決策樹（不知道該選哪個？） 
這條訊息對我現在重要嗎？
  ├ 是 → 🟢 active
  └ 否 → 為什麼不重要？
         ├ 它過時 / 被推翻了    → 🔴 stale
         ├ 它太長太煩、看夠了    → ⚪ hidden
         └ 暫時不關心，之後可能用 → 🟡 background
整理過的桌面長什麼樣（範例）

假設你開了 7 個 turn 的 session：

🟢 t0: user      "幫我看 auth.py 的 bug"          ← 主任務
🟡 t1: assistant 提到 database schema 改進         ← 收進抽屜（之後再做）
🟡 t2: tool      database 結構 dump (300 行)
🔴 t3: assistant "用 OAuth refresh"                ← 後來發現是錯的
⚪ t4: tool      OAuth library 文件 (200 行)       ← 看過了，折疊
🟢 t5: user      "OAuth 不對，是 JWT"              ← 主任務修正
🟢 t6: assistant "找到 JWT bug 了"                 ← 主任務結論

你的眼睛會自然集中在 t0/t5/t6 三個 active 行——那才是「現在的主軸」。其他被你「收進桌子的不同位置」、但都還在桌上。

關於 Note：對任何訊息加一段你自己的註解（例如「這次決定後面都還在用」「這個 tool 用法是錯的」）。同樣不會改原訊息，只是提醒未來的你。

⚠ MVP 限制：目前所有標記只影響 hub UI，不影響 model 的行為。換句話說：你 stale 一條訊息，model 下次 resume 還是會看到它——因為 Pi 還沒被改成「讀 overlay → 注入 model 前降權」。這是有意的設計：先確認你會用、用法符合預期，再決定要不要動 Pi 端讓標記真的有「執行效果」。
